# Tests

This directory holds all automated tests for the project. Production code stays under `js/`; tests mirror that layout under `tests/unit/`.

## Quick start

```bash
npm install          # once — installs firebase devDependency for optional imports
npm run test:unit    # 220 tests across 199 suites (~3s)
npm test             # all unit tests
```

Requires **Node.js 18+** (built-in `node:test` runner). No build step.

## Structure

```text
tests/
  README.md
  setup/                    Shared helpers, mocks, and generator
    test-utils.js           Family-tree test helpers
    module-test-utils.js    Export assertions and import helpers
    browser-globals.js      window / document / storage mocks
    generate-module-tests.mjs
  unit/                     One *.test.js per source module under js/
    admin-stats/
    backup-sync/
      components/
      mappers/
      services/
    constants/
    form/
    locales/
    pages/
    services/
      family-tree-inference/
        inference-engine.test.js   # hand-written (full behavioral)
    ui/
    utils/
    validation/
    app-init.test.js
    login-init.test.js
```

Every file under `js/` has a matching test under `tests/unit/` with the same relative path:

`js/form/form-state.js` → `tests/unit/form/form-state.test.js`

## Test strategies by module type

| Module type | Strategy | Example |
|-------------|----------|---------|
| **Pure logic** | Dynamic import + export checks + behavioral tests | `utils/`, `validation/`, `search-service.js` |
| **Firebase-bound** | Static source checks for expected exports (avoids App Check in Node) | `firestore-service.js`, `pages/*.js` |
| **Side-effect entrypoints** | Import without throwing | `landing-page.js`, `role-ui-sync.js` |
| **Hand-written suites** | Full positive / negative / edge coverage | `inference-engine.test.js` |

## Behavioral test conventions

Generated behavioral tests use nested `describe` blocks:

```text
describe('js/utils/birthday-date-utils.js')
  it('loads and exposes exports')
  describe('behavioral cases')
    describe('positive cases')    // happy path
    describe('negative cases')    // invalid input, rejection
    describe('edge cases')        // boundaries, empty state, idempotency
```

## Setup folder

| File | Purpose |
|------|---------|
| `test-utils.js` | `buildTestGraph`, `totalNodeCount`, `accountedMemberCount`, `clearInferenceCache` |
| `module-test-utils.js` | `assertHasExports`, `assertModuleLoads`, `importModule` |
| `browser-globals.js` | Minimal browser globals for Node |
| `generate-module-tests.mjs` | Regenerates `@generated` scaffold tests |

### Adding a new source file

1. Create `js/your-module.js`
2. Run `npm run test:generate` — creates `tests/unit/.../your-module.test.js`
3. Add behavioral tests to `BEHAVIORAL` in the generator if the module has pure logic
4. Run `npm run test:unit`

Hand-written tests are **preserved** — the generator skips files that do not contain `@generated by tests/setup/generate-module-tests.mjs`.

### Importing shared helpers

From `tests/unit/services/family-tree-inference/`:

```js
import { buildTestGraph, clearInferenceCache } from '../../../setup/test-utils.js';
```

Depth varies by test location; count `../` segments from the test file to `tests/setup/`.

## Naming conventions

- Test files: `*.test.js` suffix (e.g. `form-state.test.js`)
- Runner: `describe` / `it` from `node:test`
- Assertions: `node:assert/strict`

## npm scripts

| Script | Description |
|--------|-------------|
| `npm test` | All unit tests |
| `npm run test:unit` | Same as `npm test` (~220 tests) |
| `npm run test:generate` | Regenerate module scaffolds from `js/` |

## Direct Node invocation

```bash
node --test "tests/unit/**/*.test.js"
node --test tests/unit/utils/birthday-date-utils.test.js
```

Quoted globs are passed to Node (not the shell), which avoids zsh `nomatch` errors on empty patterns.

## Dependencies

`package.json` lists `firebase` as a **devDependency** for modules that can be imported in Node. Firebase-bound page and service tests use static export checks instead of runtime Firebase initialization to keep the suite fast and reliable.
